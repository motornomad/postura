Step 1 — Receive & Parse (changed files only)
1
Webhook Layer
Receives GitHub events, verifies HMAC signature, classifies event type (push / PR / dep update), computes the minimal change scope.

→ ChangeScope: changed files, affected modules, transitive dependents
2
Ingest Layer
Parses only changed files. Extracts code structure, runs static analysis tools, scans dependencies. Collects data — does not reason.
EXTRACTS:
Functions, classes, call graph, imports, HTTP endpoints, auth decorators, data store access, SQL read/write, PII heuristics
RUNS:
Semgrep (SAST), Bandit (SAST), pip-audit (CVEs), config scan (secrets, CORS, debug flags)
ast_parser.py endpoint_extractor.py sast_runner.py dep_scanner.py config_analyzer.py
→ StructuredIngestResult: AST nodes, call edges, endpoints, findings, dep vulns, data accesses, file imports
StructuredIngestResult flows down
Step 2 — Build & Update the Threat Graph
3
Graph Engine
Maintains the persistent threat graph in Neo4j. On each commit: identifies the affected subgraph, soft-deletes stale nodes, rebuilds from fresh ingest, computes a diff. The graph is the living map of your attack surface.
7 NODE TYPES
:Service  :Endpoint  :Function  :DataStore  :Dependency  :Finding  :TrustZone
10 EDGE TYPES
:CALLS  :HANDLED_BY  :BELONGS_TO  :IN_ZONE  :READS_FROM  :WRITES_TO  :USES  :AFFECTS  :TRUSTS  :CHAINS_TO
INCREMENTAL UPDATE
1. Snapshot affected subgraph
2. Mark stale
3. Re-ingest changed files only
4. Rebuild subgraph
5. Compute GraphDiff

The Core Insight — Why This Isn't Just Another Scanner
Static tools (Semgrep, Bandit) find individual vulnerabilities in isolation. A SQL injection gets the same severity score everywhere — doesn't matter if it's behind 3 layers of auth or wide open to the internet.
POSTURA knows the topology. The same SQL injection behind strong auth in an internal tool? LOW. Reachable from a public unauthenticated endpoint that touches PII? CRITICAL. That's what the graph + LLM reasoning enables — contextual severity based on real architecture, not rules.
Compositional risk: Finding A (no input validation on public endpoint) —[:CHAINS_TO]→ Finding B (SQL injection in internal function) because there exists a call-graph path from A's endpoint through to B's function, which :WRITES_TO a DataStore containing PII. Static tools see two separate medium findings. POSTURA sees one critical attack chain.
